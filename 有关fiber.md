## 一、Fiber 是什么？

### 定义

Fiber 是 React 16 引入的新架构，将虚拟 DOM 从树结构改成了**链表结构**，实现了**可中断渲染**。

### 解决的问题

**React 15 的问题**：
```
同步递归渲染 → 阻塞主线程 → 用户交互无响应 → 页面卡顿
```

**Fiber 的解决**：
```
可中断渲染 → 让出主线程 → 处理用户交互 → 继续渲染 → 流畅体验
```

## 二、前置知识

### 必须掌握的四个知识点

| 前置知识 | 一句话总结 | Fiber 关联 |
|---------|----------|-----------|
| **事件循环** | 调用栈空了才执行队列 | 让出主线程的原理 |
| **虚拟 DOM** | JS 对象描述 DOM | Fiber 是增强版 |
| **Diff 算法** | 对比新旧找最小变化 | Reconcile 核心 |
| **链表** | 通过指针连接节点 | child/sibling/return |

### 事件循环核心

```
调用栈（同步代码）
    ↓ 空了
宏任务队列
    ↓ 执行完
微任务队列
    ↓ 执行完
回到调用栈
```

**Fiber 应用**：Reconcile 暂停后，调用栈空了，事件循环可以处理用户交互。

## 三、Fiber 的两个阶段

### Reconcile（协调）阶段

| 特点 | 说明 |
|-----|------|
| **可中断** | ✅ 分切片执行 |
| 做什么 | Diff 计算、标记 effectTag |
| 最小单位 | 单个 JSX 元素（一个 Fiber 节点） |
| 原子操作 | 单个 Fiber 节点的处理不会被中途打断 |

### Commit（提交）阶段

| 特点 | 说明 |
|-----|------|
| **不可中断** | ❌ 必须一次性完成 |
| 做什么 | 更新真实 DOM、调用生命周期 |
| 耗时 | 很短（纯 DOM 操作，几 ms） |

### 为什么 Commit 不能中断？

```
如果 Commit 中断：
已更新：div → Hello ✅
未更新：span → World ❌
    ↓
用户看到不完整的界面（界面撕裂）
    ↓
用户体验灾难！
```

## 四、中断机制

### 中断时机

**中断只发生在两个 Fiber 节点之间**

```
处理 Fiber A（原子操作）：
├─ 计算 props
├─ 调用 render
├─ Diff 子节点
├─ 标记 effectTag
└─ 完成！（不会中途打断）

检查中断条件：
├─ 时间切片到期（5ms）？
├─ 高优先级任务到来？
└─ 决定是否中断

准备处理 Fiber B ← 中断只在这里发生！
```

### 单个 Fiber 节点是原子操作

```
单个 Fiber 节点的处理：
├─ 不会中途被打断
├─ 必须完成所有步骤
└─ 完成后才检查是否中断
```

**不存在"渲染到一半"的情况！**

## 五、时间切片

### 时间切片机制

| 参数 | 值 | 说明 |
|-----|---|------|
| 一帧 | 16.6ms | 60fps |
| React 渲染时间 | **5ms** | 预留给渲染 |
| 其他 | 11.6ms | 用户交互、动画 |

```
一帧 = 16.6ms（60fps）
├── 用户交互、动画等 → 约 11ms
└── React 渲染 → 约 5ms

React 只用 5ms，超过就暂停
剩下的时间留给浏览器处理用户交互
```

### 时间切片 vs 优先级调度（两套机制）

| 机制 | 检测什么 | 目的 |
|-----|---------|-----|
| **时间切片** | 时间是否超时（5ms） | 防止长时间阻塞 |
| **优先级调度** | 是否有高优先级任务 | 高优先级插队 |

## 六、优先级调度

### Fiber 优先级 = 渲染优先级

```
优先级决定了：
├─ 哪个更新先 Reconcile
├─ 哪个更新先 Commit
└─ 用户先看到哪个界面变化

高优先级 → 先完成渲染 → 用户先看到
低优先级 → 可能被打断 → 稍后完成
```

### 优先级分类

| 优先级 | 类型 | 示例 |
|-------|------|-----|
| 最高 | 用户输入 | 点击、键盘输入 |
| 高 | 紧急更新 | 悬停状态变化 |
| 普通 | 数据更新 | API 返回数据渲染 |
| 低 | 延迟更新 | 数据预加载 |

### 高优先级打断低优先级

```
低优先级渲染（列表）正在进行：
├─ Reconcile 切片 1 → 暂停
├─ 主线程空 → 用户点击按钮
├─ 检测到高优先级任务 → 打断低优先级

高优先级更新（按钮）独立流程：
├─ Reconcile（按钮状态变化，5ms）
├─ Commit（更新按钮 DOM，2ms）
└─ 用户看到按钮变化！（7ms）

恢复低优先级渲染（列表）：
├─ 继续 Reconcile → 完成 → Commit
└─ 用户看到列表变化（稍后）
```

**关键**：高优先级有自己独立的 Reconcile + Commit 流程！

## 七、Fiber 对象结构

### 链表连接

```javascript
const fiber = {
  type: 'div',           // 节点类型
  key: null,             // key
  child: Fiber | null,   // 第一个子节点（向下 ↓）
  sibling: Fiber | null, // 下一个兄弟节点（向右 →）
  return: Fiber | null,  // 父节点（向上 ↑）
  effectTag: 'Update',   // 需要做的操作（增/删/改）
};
```

### 遍历方式（深度优先）

```
从根节点开始
  ↓ 有 child？
  → 有：进入子节点
  → 无：看 sibling
      → 有：进入兄弟
      → 无：回到 return（父节点）
```

### Fiber 节点数量

```
有多少 JSX 元素，就有多少 Fiber 对象

不是"一个 Fiber"，而是"一个 Fiber 树"
不是"动态创建"，而是"遍历处理已存在的 Fiber 树"
```

## 八、Scheduler 和事件循环

### 三者关系

```
更新任务（带优先级）
    ↓
Scheduler（调度器）
├─ 接收所有更新任务
├─ 按优先级排序
├─ 控制时间切片
└─ 决定什么时候执行
    ↓
事件循环
├─ 执行 Scheduler 安排的任务
└─ 处理用户交互
    ↓
Reconcile
├─ Diff 计算
└─ 可中断、可恢复
```

### 事件循环不会停止

```
事件循环永远在运行：
while (true) {
  if (调用栈空了) {
    执行队列任务;
  }
  // 调用栈不空？继续等待
}

只会"等待"或"执行"，不会"停止"
```

## 九、Reconcile 的任务类型

### Reconcile 是宏任务

| 任务类型 | Reconcile 是？ |
|---------|---------------|
| 同步任务 | ❌ 不是 |
| 微任务 | ❌ 不是 |
| **宏任务** | ✅ **是！** |

### 为什么是宏任务？

```
宏任务特点：
├─ 每个宏任务之间，事件循环可以处理其他任务
├─ 用户点击、动画可以在宏任务之间被处理
└─ 完美契合 Fiber 的"中断让出主线程"需求

对比：
├─ 同步任务：一次性执行完，无法中断 → React 15 会阻塞
├─ 微任务：不会让出主线程 → 还是会卡
└─ 宏任务：切片间，能处理用户交互 → 流畅！
```

### React 使用 MessageChannel 创建宏任务

```javascript
// React Scheduler 核心代码（简化）
const channel = new MessageChannel();
const port = channel.port2;

// 收到消息时执行（这是宏任务）
channel.port1.onmessage = () => {
  // 执行 Reconcile（最多 5ms）
  performWork();
  // 没做完？安排下一个宏任务
  if (hasMoreWork) {
    port.postMessage(null);
  }
};
```

### 为什么用 MessageChannel？

| 方式 | 延迟 | React 选择？ |
|-----|------|-------------|
| setTimeout | 最小 4ms | ❌ 太慢 |
| **MessageChannel** | 几乎 0ms | ✅ **首选** |
| requestIdleCallback | 不稳定 | ❌ 兼容差 |

### 低优先级渲染流程

```
触发更新
    ↓
Reconcile（可中断）
├─ 切片 1（5ms）→ 暂停 → 事件循环处理用户交互

├─ ...
└─ 切片 N → 完成

    ↓
主线程空 → 用户点击按钮
    ↓
检测到高优先级 → 打断低优先级
    ↓
高优先级 Reconcile + Commit（7ms）
    ↓
用户看到按钮变化
    ↓
恢复低优先级 Reconcile → Commit
    ↓
用户看到列表变化（稍后）

## 十一、面试回答模板

### 问：什么是 Fiber？

> Fiber 是 React 16 引入的新架构，将虚拟 DOM 从树结构改成链表结构，实现了可中断渲染。它解决了 React 15 同步渲染阻塞主线程的问题，让用户交互能在渲染间隙被处理，提升用户体验。

### 问：Fiber 的两个阶段？

> Fiber 分为 Reconcile（协调）和 Commit（提交）两个阶段。Reconcile 是 Diff 计算过程，可以中断；Commit 是真实 DOM 更新过程，不可中断。Reconcile 可中断是因为它只是 JS 计算，Commit 不可中断是为了保证界面一致性。

### 问：什么是时间切片？

> 时间切片是 Fiber 的核心机制，React 每执行 5ms 就检查是否需要暂停，如果需要就让出主线程，让事件循环处理用户交互，然后继续渲染。这样保证了渲染期间用户交互能被响应。

### 问：Fiber 的优先级是什么？

> Fiber 优先级本质是渲染优先级，决定了哪个更新先完成 Reconcile + Commit。高优先级任务（用户输入、悬停状态）可以打断低优先级任务（数据渲染），先完成渲染让用户先看到变化。

### 问：中断发生在什么时候？

> 中断只发生在两个 Fiber 节点之间。单个 Fiber 节点的处理是原子操作，不会被中途打断。处理完一个 Fiber 节点后，React 会检查时间是否超时或是否有高优先级任务，再决定是否中断。

### 问：Reconcile 是什么任务类型？

> Reconcile 被安排为宏任务执行。React 使用 MessageChannel 创建宏任务，延迟最低。每个时间切片是一个宏任务，切片之间事件循环可以处理用户交互。

## 十二、核心要点速记

### 一句话总结

```
Fiber = 链表结构的虚拟 DOM + 可中断渲染
```

### 关键数字

| 数字 | 含义 |
|-----|------|
| **5ms** | 时间切片长度 |
| **16.6ms** | 一帧（60fps） |

2. **中断只在两个 Fiber 节点之间**

4. **高优先级打断低优先级，先完成先让用户看到**
5. **Reconcile 是宏任务，通过 MessageChannel 实现**

```

### 问题 6：Reconcile 被安排为什么类型的任务？为什么选择这种任务类型？

**我的回答**：
> 安排为宏任务，因为宏任务可以在执行间隙处理其他的任务

**得分：85/100** ✅

**正确的点**：
- 安排为宏任务 ✅
- 宏任务间隙可处理其他任务 ✅

**需要补充**：

| 任务类型 | 特点 | 适合 Fiber 吗？ |
|---------|------|---------------|
| 同步任务 | 一次性执行完，无法中断 | ❌ 会阻塞 |
| 微任务 | 不会让出主线程 | ❌ 还是会卡 |
| **宏任务** | 每个宏任务之间可处理其他任务 | ✅ 完美契合 |

**标准表述**：
> Reconcile 被安排为**宏任务**。选择宏任务是因为：
> - **同步任务**：一次性执行完，无法中断（React 15 就是这样，会阻塞）
> - **微任务**：会在当前任务后立即执行，不会让出主线程
> - **宏任务**：每个宏任务之间，事件循环可以处理其他任务（用户交互、动画），完美契合 Fiber 的中断机制
>
> React 使用 **MessageChannel** 创建宏任务，因为它的延迟最低（几乎 0ms），比 setTimeout（最小 4ms）更高效。
