### Q1: 什么是闭包？有什么用？
**什么是闭包：**
闭包是指有权访问另一个函数作用域中的变量的函数，即当函数嵌套时，内部函数引用外层函数作用域中的变量，并且内部函数被返回或者传递到外部，当外层函数执行完毕后，外层函数的作用域不会被js的垃圾回收机制回收
**有什么用：**
闭包的应用有 数据私有化，防抖节流，柯里化

### Q2: 什么是原型链？
每个对象都有一个__proto__指向原型对象，当访问对象属性的 时候，会先查找自身，如果没找到对应属性则沿着__proto__进行向上查找，直到找到或__proto__指向null，这条查找链就是原型链

### Q3: JavaScript 中的 this 指向规则？绑定方式？
this的指向取决于函数的调用方式，其中分为四种绑定方式，按优先级来排序依次是 new绑定，显式绑定，隐式绑定，默认绑定，其中new绑定是指向新创建的对象，显式绑定是指向 call apply bind中显式声明要指向的对象，隐式绑定是指向调用函数的对象，默认绑定在严格模式下是指向undefined，非严格模式下是指向window，箭头函数没有自己的this，this指向继承外部作用域

### Q4: useState是怎么存储状态的？为什么不能在条件语句中使用？
react中useState使用链表存储状态，在第一次渲染的时候，react会按顺序为每一个useState创建一个链表节点，第二次渲染会使用一个指针按顺序读取数据，第几个执行的useState就指向第几个节点的数据，因此在语句中使用useState会导致useState有时候执行有时候不执行，这样会导致指针的指向出现错误，使其他的useState取出错误的数据或者指针指向不存在的节点

### Q5: 为什么在你的项目中使用Zustand？Zustand和redux有什么区别？为什么选择 Zustand 而不是 Redux？
Zustand 比起 Redux 更加轻量化，代码量更小。Redux 使用时需要先用 Context API 创建传送通道，然后用 Provider 包裹组件，子组件才能通过 useContext 获取 Store。而 Zustand 将 Store 包装为独立的 JavaScript 对象，可以直接在组件中使用 hook 函数调用数据，因此 Zustand 比 Redux 使用更方便。

### Q6: 为什么 Redux 必须用 Provider 包裹组件，而 Zustand 不需要？
因为redux会使用Context API创建一个传输通道，只有被Provider包裹的组件才能使用useContext获取对应的Store，而zustand将Store包装为一个独立的JS对象，组件可以使用hook函数调用访问
   
一句话总结版: 
Redux 用 Context API 传递 Store，不包裹就拿不到，所以必须包裹。Zustand 的 Store 是独立对象，不依赖 Context，不需要包裹。

### Q7: Zustand中create有什么作用？
zustand的create的主要作用是接受一个回调函数，创建Store对象，返回一个函数供组件当作hook使用，其中回调函数可以接受set，get两个参数，set可以改变Store对象的状态，get可以获取当前Store对象的状态，回调函数会返回初始状态对象，这个对象会被存入Store作为初始值

### Q8: 什么是selector？有什么作用？Zustand 怎么优化性能？Zustand 怎么实现精准订阅？
selector是zustand的精准订阅机制，zustand会在内部调用selector来对被订阅的状态进行新旧值对比，只有新旧值不同才会触发组件的重渲染，从而避免不必要的性能开销

### Q9: pending 时为什么要"包装一层"？
pending需要包装一层，是因为需要将包装后的，包含resolve的回调函数压入旧promise的任务队列中，这样才能在旧promise状态发生变化的时候才能执行新promise的resolve，这样才能完成链式调用

### Q10: then 只传一个参数时，错误会怎样？为什么会这样？
then 只传一个参数时，错误会穿透到下一个 catch。因为 then 内部会设置默认的失败回调：(reason) => { throw reason }，它会抛出错误，让错误传递给下一个 then 或 catch 处理。

### Q11: React Query 怎么实现请求去重
React Query 用 queryKey 记录 请求状态 。当相同 queryKey 的请求发现 请求状态 为 pending，会复用已有的 Promise，实现去重。
